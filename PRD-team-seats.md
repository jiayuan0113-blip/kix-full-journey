# [PRD] KiX 商家端 · 多账号团队席位（Team Seats）

> 日期：2026-07-13　｜　产品：KiX 商家端 portal 后台　｜　负责人：Joyce
> 状态：设计定稿待研发评审

---

## 0. 一句话
让**一起管这门生意的人**（合伙人 / 夫妻档 / 店长 / 记账的人）都能各自登录同一个品牌后台、**共同看到全部数据、能一起干活**——补上"账号被单一登录凭证锁死、第二个人根本进不去"的真空。

---

## 1. 问题（Problem）

| 维度 | 事实 |
|---|---|
| **用户是谁** | 不是收银店员，而是**合伙人 / 夫妻档 / 店长 / 记账的人**——一起对这门生意负责的人 |
| **痛点** | 账号用**单一登录凭证唯一标识**（国内=手机号 / 海外=邮箱），只有那一个凭证能收验证码登录 → 第二个人进不去 |
| **今天的土办法** | 共用同一手机/邮箱、转发验证码 → 码只发到一处、两人不能同时用，必然卡 |
| **为什么没解决** | 登录被单凭证锁死，后台没有"再加一个登录人"的口子 |
| **他们真正要的 job** | 「让和我一起管生意的人也能登录、看到**全部**数据、能一起干活」= **平权访问**，不是受限收银 |

---

## 2. 目标（Goal）
成功 = **一个品牌账号下，多个负责人各自用自己的凭证登录，都看到同一份完整数据、都能操作日常经营**；账号的钱与成员管理仍收敛到一个人，不出安全事故。

## 3. 非目标（Non-goals）
- ❌ **不做"收银店员只能核销"的受限角色**（v1）：没有需求证据，且与"共同看到所有数据"冲突。留到有人真提再加第三个 `Cashier` 角色。
- ❌ **不做多品牌切换 / 代运营工作台**：这是"一号多人"（团队席位），不是"一人多号"。
- ❌ **不做角色升降 / 转让 owner**（v1，YAGNI，留待办）。
- ❌ **席位不收费**（见 §6）。

---

## 4. 方案（Solution）

### 4.1 角色（只留 2 个）
```
Owner（账号创建者）  = 全部权限 + 账单/换卡 + 增删成员
Member（受邀全权成员）= 看全部数据 · 核销 · 建/改/上线/下线 活动与游戏（含花钱）· 召回老客
                       仅【账单/换卡】和【成员管理】留给 Owner
```

**权限矩阵**

| 能力 | Owner | Member（全权） | 〔Cashier 只核销〕 |
|---|---|---|---|
| 看全部数据（报表/趋势/各店/新客回头客） | ✅ | ✅ | v2 按需 |
| 扫码/输码核销 | ✅ | ✅ | v2 |
| 建/改/**上线**/下线 活动 · 建/改/上线 游戏（含按到店计费花钱） | ✅ | ✅ | ❌ |
| 召回老客 · 门店管理 · 品牌素材 | ✅ | ✅ | ❌ |
| **账单与套餐 / 换绑卡** | ✅ | ❌ | ❌ |
| **邀请 / 移除成员** | ✅ | ❌ | ❌ |

> Member 看到的是**和 Owner 几乎一样的完整后台**，只是「我的」里没有【账单】和【团队管理】两块。全权成员**能自己上线活动 = 能花钱**（Joyce 2026-07-13 定：合伙人共同管生意，不互相掉链子）。

### 4.2 身份模型（研发核心改动）
现状 `Account` 直接用单一登录凭证唯一标识、一凭证一账号。改成 **人 / 归属** 分离：

```
User        id, login_type(email|phone), login_id, region
            # 登录凭证 = 海外邮箱 / 国内手机号（二选一，按区域）；login_id 唯一标识“一个人”
Membership  id, user_id, account_id, role(owner|staff→full),
            status(invited|active|removed), default_outlet_id?, joined_at
Account     去掉直接挂登录凭证 → owner 关系变成一条 role=owner 的 Membership
InviteToken id, account_id, role=member, token, expires_at,
            one_time(bool), used_count, created_by, revoked
```

- **登录凭证按区域**：**国内 = 手机号 + 短信验证码**；**海外 = 邮箱 + 邮件验证码**（+ 现有 Google/Apple SSO 可延续）。验证码发到对应渠道；后端 `auth/start` 依 region 决定渠道。
- **迁移**：现有每个 `Account` 的登录凭证 → 建 1 个 `User` + 1 条 `owner` Membership，行为不变。
- **登录 verify 后**：查该 user 的 active memberships → **0 条**=新号走注册；**1 条**=直接进（今天的行为）；**≥2 条**=出一个**轻量账号选择页**（仅多归属时才出现，不做常驻切换器——守住"团队席位≠多品牌"定位）。多归属只发生在"一个人被两家店都邀请"这种边缘。

### 4.3 邀请与加入流程
1. Owner 「**我的 → 团队 → + 邀请成员**」→ 弹出**邀请二维码 + 可复制链接**（默认 7 天有效、可再生成使旧的失效；可选一次性 or 常驻）。
2. 受邀人扫码/开链接 → 落「**加入 {店名}**」页 → **按其区域走邮箱码 / 手机验证码** → 加入成功 → 进后台。
3. 已是别账号成员的凭证 → 追加一条 membership，下次登录出账号选择页。

### 4.4 团队管理界面（「我的」新增「团队」面板 = Joyce 说的"新增登录账号设置"）
- 成员列表：头像 + 名字/凭证尾号 + 角色标签 + 状态（已加入 / 邀请中）。
- 每行可「移除」；**移除即吊销其 token/session**。
- **最后一个 Owner 不可被移除**（账号至少留一个 Owner）。
- v1 不做角色升降 / 转让 owner。

### 4.5 门店归因（保住到店数据准确）
- 邀请成员时**可选填默认门店**（单店商家自动 = 唯一门店，全程无感）。
- 成员核销时 `outlet_id` = 其默认门店；未指定则核销页顶部让其选当前门店。

---

## 5. 给研发的安全红线
- 后端按 `membership.role` 鉴权，**不能只靠前端隐藏**：Member token 调【账单】【成员管理】接口一律 **403**。
- 邀请 token：有有效期 + 可撤销 + 用量上限（一次性/常驻可选）。
- 移除成员 → 即时吊销其 session/token。
- 登录渠道按 region 分流，验证码防滥用沿用 §12 现有分层防护（限流 / 防短信轰炸）。

---

## 6. 计费
**席位免费、不限数**。canon「软件永久免费」，计费只按品牌 MAU / walk-in；席位收费会挡住"让合伙人上岗"这个正向动作。

---

## 7. 成功指标（leading）
- 开通团队的账号里，**≥2 个 active member 的账号占比**（功能被真用）。
- 被邀请人**从收到链接到成功登录的完成率**（邀请流畅度）。
- 多成员账号 vs 单成员账号的**留存 / 活动上线数**差异（协作是否带来更多经营动作）。

## 8. 待办 / 开放问题（按重要性）
1. **Cashier 只核销角色**：等有商家真提"店员乱看数据/改价"再加，作为第三角色。
2. **角色升降 / 转让 Owner**：v1 不做，观察是否高频。
3. **SSO（Google/Apple）与验证码凭证的账号合并规则**：海外同一人用邮箱码 vs Google 登录，是否归并为同一 User——需研发确认现有部署逻辑。
4. **部署版对齐**：`partner-new.letskix.com` 现有邮箱登录反而符合"海外=邮箱"，需和本地主线一起把 region 分流讲清。
