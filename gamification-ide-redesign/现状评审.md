# [评审] KiX Gamification IDE（AI 建游戏工作台）— partner-test 现状问题清单

> 日期：2026-06-30
> 评审对象：`https://partner-test.letskix.com/kix-gamification-ide/?order_id=…&brand=5`
> 形态：左 AI 助手（自然语言改游戏 + 快捷 chip）｜中 9:16 手机预览（Supermarket Fruit Crush 三消）｜右 Assets 素材面板｜顶栏品牌标题 + 撤销/重做/历史 + Edit text/image + 中/EN + Save
> 评审方法：第一性原理框定「这页唯一的活」→ 设计-现实 gap，按 HIG + 产品 canon + 业界一流 AI builder（Lovable / Bolt / Canva / Figma）对照，按严重度排序。

---

## 第一性原理：这页唯一的活是什么

**一句话**：让商家用最少操作，把一个现成玩法模板「改成自己品牌的样子」，并相信预览里那台手机 = 客人将看到的成品。

**充要条件**（任一不满足这页就失败）：
1. 预览 = 真成品（所见即所得，零穿帮）。
2. 改的入口唯一且清楚（一个主动作：告诉 AI 改什么 / 直接点元素改）。
3. 商家看不到任何「不是给他看的东西」（内部标签、代码、未翻译串）。

**问题类型**：COVERAGE（多处要收，不是单根杠杆），但 **P0 全部命中条件 3——穿帮**，这是当前最伤的一类。

---

## P0 — 穿帮级（partner demo / 上线前必须先修）

### P0-1　手机预览里漏出中文：计时标签 `慢悠悠`（EN 模式下）
- **现象**：EN 模式，游戏预览顶部计时器显示 `慢悠悠 58`（应为时钟图标 / "TIME"）。DOM 里另有 `慢慢来` 同类串。
- **为什么最严重**：漏中文的位置不是后台按钮，是**手机预览 = 商家要拿去给自己客人看的成品**。这一处穿帮直接击穿条件 1（预览=真成品）和"把路过的人变回头客"的信任前提。
- **机制**：round-time 档位的中文文案（"慢悠悠"= 慢节奏）被当作 UI label 直接渲染，没走 i18n；中/EN 切换没有驱动预览内的运行时文案。
- **修**：① 计时器 label 用图标或 locale 串，不要把档位描述词当 label；② 中/EN 切换必须同时切运行时（游戏内）文案，不只切外壳 chrome。
- 置信度：高（截图实证）。

### P0-2　未压缩 JS 把中文源码注释打包给了客户端
- **现象**：客户端 bundle 里夹带大量中文开发注释，例如 `避免被当成可编辑选中而弹工具条`、`让编辑器在素材面板选中对应文件`、`首批渲染完成后这些重扫即停`、`蓝框与浮动工具栏跟随` 等数十条。
- **为什么是问题**：(a) 内部实现/思路 IP 直接暴露给任何 partner / 竞品；(b) 说明 partner-test 没做 minify，prod 多半同样；(c) 白白增大体积、拖慢首屏。
- **修**：构建走 minify + strip comments；至少对外环境去注释。
- 置信度：高（DOM grep 实证）。注：这些是 JS 注释、用户肉眼看不到，但属可被任何人 view-source 拿到。

### P0-3　快捷 chip 上漏出占位标签 `Deep`
- **现象**：chip "Match Tile Pictures" 和 "Round Time" 各挂一个圆角 pill `Deep`（`<span class="accent-ring…">Deep</span>`）。
- **为什么是问题**：`Deep` 对商家零语义，是内部分类/难度 token 漏出。商家会问"Deep 是什么"——加了控件却不解决任何用户问题（违反"加控件前自检它此刻解决用户什么"）。
- **修**：要么去掉这个 pill，要么换成商家看得懂的状态词（如"已改/默认"），并解释它代表什么。
- 置信度：高（DOM 实证）。

## P1 — 焦点/信息架构（违反"一页只做一个动作"）

### P1-1　两个并列底部 CTA = 没有焦点
- **现象**：左面板底部 `✦ Chat with AI` 与 `✦ Not sure what to change? Try here!` 两枚都带 ✦、都指向 AI 对话。
- **为什么是问题**：canon 明令"两个并列强调 = 没有焦点"。同一动作两个入口，商家要先在两者间做无意义选择。
- **修**：合成一个。"Not sure what to change?" 作为 AI 输入框的 placeholder/副标即可，不另起一枚按钮。
- 置信度：高。

### P1-2　三个标题抢主位
- **现象**：头部同时出现 `KiX Game Editor` / `Make it your own` / `Supermarket Fruit Crush`。
- **为什么是问题**：没有单一标题锚点，商家不知道"我现在在哪、改的是哪个东西"。一流 builder（Figma/Canva）头部只给一个：当前文件名。
- **修**：主标题 = 当前游戏名（Supermarket Fruit Crush，可点改名）；"Make it your own"降为副标或去掉；"KiX Game Editor" 收进 logo 区。
- 置信度：中-高。

### P1-3　chip 的副值与当前游戏不符（模板默认没同步）
- **现象**：游戏是水果/超市三消，但 chip 副值是 `Coffee Items`（Match Tile Pictures 下）、`Shorter`（Round Time 下）。
- **为什么是问题**：`Coffee Items` 与当前 brand=5 的水果主题无关，疑似模板默认值没按实际游戏/品牌回填，商家会以为点了会变成咖啡。击穿"预览=真成品"的预期一致性。
- **修**：chip 副值必须反映当前游戏的真实当前态（或写成动作"换成我的商品图"而非写死某品类）。
- 置信度：中（需研发确认 chip 副值的取数来源）。

## P2 — HIG / 交互细节

### P2-1　窄宽度下两侧面板整体消失，且无召回入口
- **现象**：~900px 宽时，左 AI 面板 + 右 Assets 面板全部隐藏，只剩中间预览 + 两枚底部 CTA。
- **为什么是问题**：商家在小窗/平板上会"丢失所有编辑能力"且没有明显办法把面板调回来。若目标设备只有桌面则降级为低优。
- **修**：定义断点行为——窄屏把面板收成可点开的抽屉/Tab，而不是直接消失。
- 置信度：中（取决于目标设备，建议确认商家实际用机）。

### P2-2　`Save` 与发布闸门关系不清
- **现象**：顶右主按钮是绿色 `Save`。
- **为什么要确认**：产品 canon 的硬闸门在 **publish**（且注册后置到发布）。这里 `Save` 是"暂存草稿"还是"提交上线/发布"？若 `Save` 实为发布，命名误导；若只是暂存，那发布入口在哪、注册闸门在哪一步要讲清。
- **修**：区分"暂存草稿"(Save) 与"发布/上线"(Publish) 两个动作与文案；发布点对齐注册闸门。
- 置信度：中（需研发确认 Save 的真实语义）。

### P2-3　Edit text / Edit image 模式态
- **现象**："Edit text → Click to start, then click the text you want to change" 是显式模式切换。
- **判断**：模式化直接点改本身 OK（Canva 同款），但要保证：进入模式有明确视觉态、退出顺手、与"AI 对话改"两条路径不打架。当前未见明显冲突，**列为观察项**，非缺陷。
- 置信度：中。

## 因果链根因

- **根因 1（穿帮类 P0-1/2/3 同源）**：i18n 与构建产物治理缺位——运行时文案没全量走 locale、bundle 没 strip。→ 修根因 = 建立"对外环境零中文穿帽 + 强制 minify"的构建/发布门禁，P0-1/2/3 一起消解。
- **根因 2（焦点类 P1）**：页面按"把所有能力都摆出来"组织，而非按"一个主动作"组织。→ 修根因 = 重排为单焦点（一个标题 + 一个改入口 + 预览），P1-1/1-2 一起收。

## 创意打分（本评审）
- S1 具体 3（点名具体串/类名/像素宽度）/ S2 机制 2（i18n 未驱动运行时、构建未 strip）/ S3 先例 2（Figma/Canva/Lovable 对照）/ S4 可证伪 3（每条都可截图复核）= **10/12 → REAL_INSIGHT**

## 结论
Trinity converged, 1 round。可直接转研发的优先级：**先清 P0 三条穿帮（上线/demo 阻断）→ 再收 P1 焦点 → P2 按目标设备与发布语义确认后处理。**
