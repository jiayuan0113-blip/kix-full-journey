## 1. 执行摘要

为**非技术 F&B 街边小店主**重构商家后台数据页的「游戏」tab：从"只展示当前在跑游戏的单聚合数据"，改为**「概览（含已下线游戏的横向对比，可点）→ 单游戏历史详情」两层结构**，让商家在**游戏下线后仍能查看它上线期间攒下的历史数据**、并**按单个游戏维度看/对比**。核心口径修正：游戏玩次 = **该游戏对象的全部游玩（独立上线 + 在活动里被玩）**，下线不删数据。

---

## 2. 问题陈述

### 谁有这个问题
账号下建过 ≥1 个游戏、且会换游戏/下线旧游戏的商家（尤以爱试不同玩法的店主，如奶茶店 Ah Keong 换过 3–4 个游戏）。

### 问题是什么
1. **下线游戏的历史数据消失**：游戏一下线（状态回 draft），数据页立刻看不到它上线期间的任何数据。
2. **不能按单游戏维度深看**：原设计只有一个跨游戏聚合看板，看不了单个游戏的完整表现（玩家/时长/每日趋势）。

### 为什么痛（机制层，非"体验不好"）
数据挂在 **"游戏当前是否 live 的瞬时态"** 上（`GAME_METRICS` 单聚合 + `hasLiveGame` 闸门），而非挂在 **"游戏对象的生命周期累计"** 上。商家的经营记忆不该随对象状态归零——数据所有权属于商家，应与"游戏此刻是否在跑"解耦。

### 证据 / 依据
- Joyce 反馈（2026-07-14）：「游戏下线了，用户其实也关心旧数据，想看过去的数据」。
- 业界一致铁律（三体 ① grounding）：Google Ads（暂停/已删 campaign 历史全留、靠状态筛选 + 单对象下钻）、Mailchimp（archive 保留 report）、Shopify/Mailchimp（per-product/campaign 明细为默认下钻单位）——归档/暂停对象的历史数据**永远保留 + 可查**，从不靠删。

## 4. 战略背景

- 契合数据页 IA canon（2026-07-13 三体）：**参与度（玩次/时长，跟游戏走）→ Games tab；转化（到店/券/ROI，跟活动走）→ Activities tab**。本次把"参与度跟游戏对象走"落到实处。
- 守 canon：游戏状态机**只有 draft/live，不引入第三态**；单一焦点 + 加法伤留存（本次加法属"填事实真空"的正当豁免——下线游戏历史全站无处可达）。

## 5. 方案总览

### 5.1 状态与口径（先认这些）
- **游戏状态机不变**：`draft` / `live`。数据层用 `status:"offline"` 表示"**曾上线过、现已下线**"（= 纯展示态 + `everLive` 语义，**非游戏状态机态**）。下线**不删数据**。
- **玩次含活动内游玩**：某游戏玩次 = 独立上线被玩 + 在绑定它的活动里被玩（同一游戏对象累计）。**与 Activities tab 漏斗顶端「扫码玩」有意重叠**（两 tab 回答不同问题：参与度 vs 转化）。文案注明「包含在活动里玩的 / including plays inside activities」。**推翻**原"独立上线纯玩数据/无奖品"口径。

### 5.2 目标态流程
```
数据页 → 顶部分段「活动 / 游戏」→ 选「游戏」
  ├─ 从没任何游戏上线过 ......................→ 空态「去我的游戏」
  ├─ 恰好 1 个曾上线游戏 ....................→ 直接进该游戏「详情」（跳过概览，省一次点击）
  └─ ≥2 个曾上线游戏 ........................→ 「概览」
        └─ 点任一游戏（横向对比条）..........→ 该游戏「详情」→ 顶「‹ 所有游戏」返回概览
```

### 5.3 页面/区块清单
| 区块 | 出现在 | 内容 |
|---|---|---|
| **概览 Hero** | 概览 | 游玩次数总数（深色 band）+ 日均附注 + 说明「所有游戏被玩的总次数，包含在活动里玩的」+ sparkline |
| **KPI 卡 ×2** | 概览 + 详情 | 玩家数 / 平均时长；**单行卡**（标签左、数字右、垂直居中） |
| **各游戏横向对比条形图** | 概览 | 每根条 = 游戏名 + `Live`/`Offline` 标 + 值；**含已下线游戏**；**每根可点** → 详情；**维度切换 玩次/玩家/时长**；默认当前维度**降序** |
| **详情 Hero** | 详情 | 在跑=绿 hero + "上线以来" + 日均 + sparkline；**已下线=灰存档 hero**（降饱和）+ "在线期间"口径（无日均、无增长箭头） |
| **已下线提示条** | 详情（仅 offline） | 黄条「游戏已下线。下面是它上线期间攒下的数据。想继续收集，去我的游戏重新上线。」 |
| **每日游玩柱状** | 详情 | 在跑=近 7 天柱状；已下线=**只画在线段 + 结束竖虚线**（不画空柱）；小样本=**sparkline 降级** + 提示 |

## 6. 成功指标
- **主**：下线游戏历史数据的可达率 = 100%（曾上线有数据的游戏，100% 可在数据页查到其历史）。
- **次**：Games tab 内单游戏详情的进入率（衡量下钻被使用）；多游戏商家使用横向对比维度切换的比例。
- **护栏**：不新增弹窗/浮层（守加法伤留存）；活动 tab 与主页数据口径不因本次改动而打架。

## 7. 用户故事与需求（含边界态）

### Epic 假设
> 我们相信：把游戏数据按"游戏对象生命周期"存、并做成「概览→下钻」+ 保留下线历史，能让商家在换游戏后仍回看旧数据、判断哪种玩法有效——因为当前数据挂瞬时态、一下线就消失。验证 = 下线游戏历史 100% 可达 + 单游戏详情被使用。

### Story 1 · 概览：横向对比（含已下线、可点、可切维度）
**As** 有多个游戏的商家，**I want** 一眼比出哪个游戏最受欢迎并能点进去深看，**so that** 判断哪种玩法有效。
**验收：**
- [ ] 概览显示：Hero（总玩次 + 日均）+ 玩家数/平均时长 KPI 卡 + 各游戏横向对比条形图。
- [ ] 条形图**含已下线游戏**，每行挂 `已上线(Live)`/`已下线(Offline)` 标；**默认按当前维度降序**。
- [ ] 维度切换 `玩次 / 玩家 / 时长`，条长与数值随之变化。
- [ ] 每根条可点 → 进该游戏详情；静止态即有可点暗示（行描边 + `›`）。
- [ ] **不显示**"每天游玩次数（所有游戏合并）"区块（与 Hero/条形图同一总数，去重）。

### Story 2 · 单游戏详情（在跑）
**验收：**
- [ ] 顶部游戏头（缩略色块 + 名 + `已上线` 标 + `上线 N 天·至今在跑`）。
- [ ] 绿 Hero：游玩次数 + `日均约 N 次` 附注 + 说明「客人玩这个游戏的总次数，包含在活动里玩的」+ sparkline。
- [ ] KPI 卡：玩家数 / 平均时长。
- [ ] 每日游玩：近 7 天柱状。
- [ ] ≥2 游戏时顶部有「‹ 所有游戏」返回；顶部日期 pill 可用。

### Story 3 · 单游戏详情（已下线，历史存档）⭐
**验收：**
- [ ] 游戏头标 `已下线` + `上线 N 天（起–止日期），已下线`。
- [ ] **黄提示条**（见 §5.3 文案）。
- [ ] **灰存档 Hero**（降饱和）：eyebrow「游玩次数 · 在线期间」，**无日均、无增长箭头**；说明「它上线的 N 天里，一共被玩这么多次」。
- [ ] KPI 卡副标改「在线期间多少人玩过」。
- [ ] 每日游玩：**只画在线段的柱** + 末尾一条 `下线` 竖虚线；**不画 0 值空柱**。
- [ ] **顶部日期 pill 隐藏**（今天/近7天/近30天 对下线游戏无意义、会全空穿帮）。

### Story 4 · 单游戏直接进详情
**验收：**
- [ ] 恰好 1 个曾上线游戏时，Games tab **跳过概览、直接进该游戏详情**，且**不显示**「‹ 所有游戏」返回（无处可返）。
- [ ] 从 1 个增加到 2 个游戏后，自动变为"概览"，该游戏成为其中一根条（其绝对数值保持醒目）。

### Story 5 · 小样本 / 稀疏
**验收：**
- [ ] 游戏 `small`（玩次低于阈值）时，详情每日柱状**降级为 sparkline** + 「数据太少，多跑几天就有了」；Hero 不显 sparkline/趋势。

### Story 6 · 空态
**验收：**
- [ ] 判据 `hasEverLive`（曾上线过任何游戏）取代 `hasLiveGame`——**只要曾上线有数据、即使现在全下线，也走概览/详情**；仅"从没任何游戏上线过"才空态「去我的游戏」。

### Story 7 · 字体 / 视觉手艺（设计评审 P0）
**验收：**
- [ ] 指标数字走**数据字重**：Hero 54px/800、KPI 36px/700、对比条 800；**全站数字 `tabular-nums`**。
- [ ] 单位（如 `s`）用 `.rh-unit` 缩小变灰。
- [ ] 不使用 Plus Jakarta Sans 900（Black）做指标数字（放大发闷）。

## 8. 数据模型 / 接口契约（研发对接）

原型 mock 在 `data.jsx` `GAME_METRICS`；研发需接真实服务，**按游戏对象累计（含活动内游玩）、下线不删历史**。

```jsonc
GAME_METRICS = {
  // 概览聚合（跨所有 everLive 游戏）
  "plays": 1304,        // 总游玩次数（含活动内）
  "players": 974,       // 去重玩家数（< 各游戏之和）
  "avgPlaySec": 36,
  "dailyAvg": 54,       // 日均（= plays / 在线天数），用于 hero 附注
  "spark": [60,66,70,78,84,92,100],   // hero sparkline 高度 0–100
  "games": [
    {
      "id": "wheel",
      "kind": "spin", "color": "#16A34A", "c": "linear-gradient(...)",
      "name": { "en": "Lucky Spin", "zh": "幸运大转盘" },
      "status": "live",              // live | offline（offline=曾上线现已下线，非游戏状态机态）
      "liveMeta": { "en": "Live for 24 days", "zh": "上线 24 天 · 至今在跑" },
      "plays": 720, "players": 512, "avgPlaySec": 41,
      "dailyAvg": 30,               // 仅 live 用
      "trend": [ { "d": {"en":"Mon","zh":"一"}, "v": 88 }, ... ]  // 在跑=近7天；下线=在线段分桶
      // "ended": true,             // offline 游戏置 true → 每日图末尾画结束虚线
      // "small": true              // 小样本 → sparkline 降级
    }
    // ...更多游戏，含 status:"offline" 的历史项
  ]
}
```

**后端硬要求：**
1. **玩次归因到游戏对象**：一次游玩无论来自独立游戏码还是活动码（活动 1:1 绑定某游戏），都累加到该 `game.plays`。
2. **`everLive` 标记 + 历史留存**：游戏首次上线打 `everLive`；下线（回 draft）**不清空**其累计数据与时间序列；数据层据此产出 `status:"offline"` 项。
3. **单游戏时间序列**：需按"该游戏在线期间"提供 `trend`（下线游戏 = 其在线那段，不含下线后的空天）。
4. **去重玩家**：聚合 `players` 为跨游戏去重，非各游戏之和。

## 10. 风险与专项（含 Stage 5 流程走查·静默失败）
| # | 风险 / 静默失败 | 处理 |
|---|---|---|
| R1 | **两 tab 玩次重叠致店主困惑**（Games 玩次 vs Activities 漏斗「扫码玩」对不上） | 有意重叠、口径不同；文案已注明「包含在活动里玩的」。研发勿"修正"成互斥。 |
| R2 | 下线游戏日期 pill 若不隐藏 → 今天/近7天全空**穿帮** | 已隐藏（offline 详情不渲染 datepills）。 |
| R3 | 下线后**再编辑重上线**同一游戏 | 同一 `game.id` 数据连续累计；在线段间的空档由 trend 自然表达。研发按 id 累计。 |
| R4 | 小样本大数字/空柱让店主觉得"惨/图坏了" | sparkline 降级 + 「数据太少」提示（阈值 `small` 可调）。 |
| R5 | 单游戏→多游戏切换时店主记的数字"变成一根条" | 概览条保留绝对数值、醒目；过渡不突兀。 |

## 11. Out of Scope（本期不做）
- **专门的多游戏并排对比视图**（Linda 的高级诉求）——靠横向对比条 + 维度切换满足即可，不新建视图（守单一焦点）。
- **给游戏引入第三态 offline**（状态机层）——用数据层 `everLive`/展示态解决，不动状态机。
- **跨账户/全站 lifetime 总览页**——属未来账户总览，不在数据页。
- 活动 tab 改动——本次不碰。
